# Problem Validation Framework

A structured framework for validating problems before investing in solutions. Use this to ensure you're building the right thing.

---

## When to Use This Skill

Use this skill when:
- You have an idea for a new product/feature
- You need to confirm a problem exists
- You want to understand customer pain points
- You're assessing market opportunity
- You need to prioritize between multiple problems

---

## Problem Validation Checklist

### 1. Problem Existence
- [ ] Problem clearly articulated in customer language
- [ ] Customers can describe the problem in their own words
- [ ] Problem occurs frequently (daily/weekly/monthly)
- [ ] Problem is painful enough that customers seek solutions
- [ ] Current alternatives are inadequate

### 2. Customer Identification
- [ ] Target customer segment clearly defined
- [ ] Early adopters identified
- [ ] Customer personas created
- [ ] Access to customers for interviews
- [ ] Sample size sufficient (min 10-15 interviews)

### 3. Market Size
- [ ] TAM (Total Addressable Market) estimated
- [ ] SAM (Serviceable Addressable Market) estimated
- [ ] SOM (Serviceable Obtainable Market) estimated
- [ ] Market growth rate understood
- [ ] Competitive landscape mapped

### 4. Willingness to Pay
- [ ] Customers currently spending on alternatives
- [ ] Price sensitivity understood
- [ ] Value metric identified (what they pay for)
- [ ] Budget owner identified (B2B)
- [ ] Purchase process understood

## Problem Interview Guide

### Opening (2 min)
```
"Thanks for talking with me. I'm researching [domain area] to understand 
how people handle [problem area]. I'd love to learn about your experience.

There's no right or wrong answer - I'm just here to learn from you."
```

### Context Questions (5 min)
```
- Tell me about the last time you [faced the problem]
- Walk me through how you handled it
- What tools/processes did you use?
- How long did it take?
```

### Problem Exploration (10 min)
```
- How often does this happen?
- How painful is this? (1-10 scale)
- What have you tried to solve it?
- Why didn't those solutions work?
- How do you currently cope?
```

### Impact Questions (5 min)
```
- How much time/money does this cost you?
- What would solving this enable you to do?
- Have you looked for solutions?
- What would an ideal solution look like?
```

### Closing (3 min)
```
- Is there anything else I should know?
- Can I follow up if I have more questions?
- Do you know others facing this problem?
```

## Problem Validation Scorecard

### Scoring System (1-5 for each)

| Criteria | Score | Evidence |
|----------|-------|----------|
| **Frequency** | 1-5 | How often problem occurs |
| **Intensity** | 1-5 | How painful it is |
| **Prevalence** | 1-5 | How many people have it |
| **Willingness to Pay** | 1-5 | Willing to pay for solution |
| **Existing Alternatives** | 1-5 | Current solutions inadequate |

### Interpretation

| Total Score | Action |
|-------------|--------|
| **20-25** | ✅ Strong signal - proceed to solution |
| **15-19** | ⚠️ Moderate signal - more validation needed |
| **10-14** | ❌ Weak signal - reconsider problem |
| **<10** | 🛑 No signal - abandon or pivot |

## Red Flags (Stop Signals)

🚩 **Customer says:**
- "That's interesting" (instead of "I need this")
- "I'd use that" (without commitment)
- "Can you show me when it's ready?"
- "My company would never pay for this"
- "This happens once a year"

🚩 **Interview patterns:**
- Can't describe problem in own words
- No current workaround exists
- Not the decision maker
- Problem only exists in edge cases
- Solution depends on behavior change

---

## Green Flags (Go Signals)

✅ **Customer says:**
- "I hate this problem"
- "I've tried X, Y, Z and nothing works"
- "How much does it cost?"
- "Can I be a beta customer?"
- "This costs me $X per month"

✅ **Interview patterns:**
- Emotional reaction when discussing problem
- Already spending money/time on workarounds
- Decision maker with budget
- Problem occurs frequently
- Actively seeking solutions
